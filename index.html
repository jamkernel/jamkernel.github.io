Arquitectura de Microkernel

Inspirado en los kernels de sistemas operativos, JAM separa el núcleo mínimo (orquestación, eventos, ciclo de vida) de los drivers (BrowserAdapter, NodeAdapter), servicios (crypto, identity, mesh) y sistema de archivos (BatchStorage). Cada componente es intercambiable sin afectar a los demás.

Único en el ecosistema P2P
🔌

Plataforma como Driver

BrowserAdapter usa WebRTC + IndexedDB. NodeAdapter usa WebSocket nativo + FS. DenoAdapter usa Deno APIs nativas. Todos comparten la misma interfaz IPlatform. El mesh, la identidad, el cifrado — todo funciona sin saber qué plataforma hay debajo. Plug and play.

Un kernel, 4 plataformas
🛡️

Seguridad Zero-Trust desde el Nacimiento

ECDSA P-256 para identidad. AES-256-GCM para cifrado. Firma digital en cada mensaje. peerId derivado criptográficamente de la clave pública: no hay suplantación posible. Purga forense de claves en RAM al cerrar sesión. Rate limiting anti-DoS. Blacklist automática. Zero-trust verdadero.

Militar grade, zero deps
📦

Un Solo Archivo. Literalmente.

No hay node_modules. No hay bundlers. No hay npm install. No hay archivos de configuración. No hay build steps. Todo el kernel, todos los adaptadores, todos los servicios, toda la criptografía, todo el servidor de señalización — en un único archivo de ~3,500 líneas. Cópialo, impórtalo, funciona.

Problemas Comunes y Soluciones

Porque incluso el mejor software encuentra obstáculos. Aquí están los más frecuentes y cómo resolverlos.

🔴 El peer no conecta al servidor de señalización

Causa: Puerto cerrado, firewall bloqueando, URL mal configurada, TLS mal emitido.
Solución: netstat -an | findstr :PUERTO para verificar. Agregar regla de firewall. Verificar que el certificado TLS coincida con el hostname.

🔴 Firma inválida / bad_signature

Causa: El peer que envía no tiene su clave pública registrada en el receptor. O ataque de suplantación.
Solución: Intercambiar claves públicas tras peer:connected. Limpiar con mesh.clearVerifiedPeers(). El servidor ya rechaza announces inválidos.

🔴 Rate limiting / attack_detected

Causa: Un peer envía más de 60 msg/s o mensajes de más de 256 KB.
Solución: Automático — el kernel blacklistea al infractor 60s. Para ajustar: kernel.mesh.MAX_MESSAGES_PER_SECOND = 120.

🔴 Mesh se desconecta al recargar el navegador

Causa: IndexedDB fue borrado o modo incógnito. La identidad ECDSA se regenera.
Solución: Exportar identidad a localStorage antes de recargar. O simplemente aceptar que es un peer nuevo (comportamiento esperado).

🔴 Cluster: workers no se comunican

Causa: IPC del proceso primario como relay. Si un worker muere, sus peers quedan huérfanos.
Solución: El primario reenvía eventos automáticamente. Para >1000 peers, considerar NATS o Redis como message broker externo.

🔴 Logs no se escriben al archivo

Causa: El directorio padre no existe o permisos insuficientes.
Solución: mkdir -p /var/log/jam antes de arrancar. El archivo se crea solo, el directorio debe existir. Rotación automática a 10MB.

🔴 TLS: certificado no aceptado en navegador

Causa: Certificado autofirmado o expirado, o hostname no coincide con el CN/SAN.
Solución: En desarrollo, aceptar excepción manual. En producción, usar Let's Encrypt: certbot certonly --standalone -d midominio.com.

🔴 El mesh no descubre peers

Causa: Los peers apuntan a diferentes servidores de señalización, o NAT bloquea WebRTC.
Solución: Verificar que todos usen la misma URL de signal. Para redes locales, signalServer: 'ws://192.168.1.100:8080'. STUN configurado por defecto.

Quick Start

De cero a mesh P2P en 30 segundos. Sin npm install. Sin configuraciones. Sin servidores externos.

# 1. Arrancar un peer Node.js con servidor de señalización + mesh
$ node jam-omni-kernel.js --room mi-sala --password "clave-segura-2026"

# 2. Abrir http://localhost:[PUERTO] en el navegador
#    El chat P2P funciona al instante
// 3. O desde código propio:
const { JAMOmni } = require('./jam-omni-kernel.js');

const kernel = await JAMOmni.createKernel({});
const identity = await kernel.whenIdentityReady();
console.log(`Peer ID: ${identity.peerId}`);

// Escuchar mensajes entrantes
kernel.events.on('peer:message', ({ senderId, message }) => {
    console.log(`${senderId}: ${message.text}`);
});

// Unirse a una sala y transmitir
await kernel.startSession('mi-sala', 'mi-clave-secreta');
kernel.events.on('peer:connected', ({ peerId }) => {
    await kernel.broadcast({ text: 'Hola mundo P2P!' });
});
# 4. Producción real: TLS + Cluster + Logging
$ node jam-omni-kernel.js \
  --port 443 \
  --tls-key /etc/letsencrypt/live/midominio/privkey.pem \
  --tls-cert /etc/letsencrypt/live/midominio/fullchain.pem \
  --room produccion \
  --password "c0ntr4s3ñ4-muy-fu3rt3-2026" \
  --cluster --workers 4 \
  --log-file /var/log/jam/mesh.log \
  --log-level info
